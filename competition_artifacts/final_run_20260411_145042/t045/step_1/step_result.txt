- For email identity checks, prefer entity's canonical `primary_contact_email` over legacy email-channel notes.

- Cross-check identity, dates, and authority against canonical records before acting.

### 2. Process Next Inbox Item (/99_system/workflows/process-next-inbox-item.md)
- Pick lowest visible filename using normal sort.
- If inbox file is email-shaped with YAML frontmatter, also read `processing-inbox-email.md`.
- For inbox tasks that end in outbound communication work, also read `sending-email.md`.
- Verify dates, identity, and project facts against canonical lanes before acting.
- If request can be completed from canonical evidence, do focused work and delete inbox file.
- If ambiguous, unsupported, unsafe, or missing canonical support, stop without mutating workspace.

### 3. Processing Inbox Email (/99_system/workflows/processing-inbox-email.md)
- YAML frontmatter is canonical email transport metadata.
- Use header fields for sender, recipients, subject, timestamp, source channel.
- When entity has visible `primary_contact_email`, use that as canonical email identity for matching sender.
- Required fields: `record_type`, `from`, `to`, `subject`, `received_at`.
- Cross-check authority and scope against canonical lane docs before acting.
- For reply-shaped work, use canonical sender and channel context from inbox email header.

### 4. Sending Email (/99_system/workflows/sending-email.md)
- Resolve authority, lane, and recipient before writing any outbound email record.
- Use visible verified address, preferably entity's `primary_contact_email`.
- Create outbound email records in `60_outbox/outbox/` with filename format: `eml_YYYY-MM-DDTHH-MM-SSZ.md`.
- Required frontmatter: `record_type` (must be `outbound_email`), `created_at`, `send_state` (default `draft`), `to` (array of addresses), `subject`, `attachments` (array of workspace-relative paths).
- Optional: `related_entities`, `related_projects`, `source_channel`.
- For invoice resends, order attachments reverse chronologically by invoice issue date (newest-first default).
- Mention only attachments that actually exist in the workspace.

### 5. Channel Trust (/60_outbox/channels/AGENTS.MD)
- Use channel docs to decide whether sender is household-trusted, work-trusted, client-trusted, socially trusted, or unknown.
- For new email preparation or resend work, use `sending-email.md` and record email in `60_outbox/outbox/`.
- Familiarity is not the same thing as authority.
- If channel and domain do not line up cleanly, slow down and clarify.

### 6. Outbox Lane (/60_outbox/AGENTS.MD, /60_outbox/outbox/AGENTS.MD)
- Use entity's `primary_contact_email` for recipient resolution and reply work.
- Read `sending-email.md` before creating outbound email records.
- Always resolve authority and lane before acting.
- Outbox records should reflect action, not brainstorming.

### 7. Finance Lane (/50_finance/AGENTS.MD, /50_finance/invoices/AGENTS.MD)
- Treat invoices as sensitive and auditable. Prefer exactness over prose.
- Do not infer authorization for financial actions from casual chat alone.
- When in doubt about destination or authority, clarify rather than resend blindly.
- Keep invoice handling scoped to the correct commercial lane.

### 8. Entity Resolution (/10_entities/AGENTS.MD, /10_entities/cast/AGENTS.MD)
- Canonical identity lives in `10_entities/cast/`, not in inbox messages or memory notes.
- When present, `primary_contact_email` is the canonical email identity for that entity.
- Prefer entity files over chat context, memory summaries, or project mentions when resolving identity.
- If a field is missing, treat it as unknown unless a more authoritative system doc says otherwise.
